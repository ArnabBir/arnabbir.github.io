TERM reliability, sre

Availability (SLO)

Fraction of requests that are served successfully within the defined correctness + latency contract. Often expressed as 'nines'. Must be paired with a measurement window and clear error definition.

Related Ch 1
TERM reliability, sre

Error budget

The allowable amount of unreliability implied by an availability SLO (e.g., 99.9% over 30 days ‚áí ~43.2 minutes of 'bad' time). Used to balance feature velocity and stability.

TERM performance

Tail latency (p99/p999)

High-percentile latency that dominates user experience in composed systems. Tail latency is amplified by queues, retries, and fanout.

Related Ch 1 Ch 3 Ch 8
TERM scalability

Load parameter

A measurable dimension of workload that predicts resource usage: QPS, write ratio, fanout, distinct keys, skew, payload size, working set.

Related Ch 1
TERM correctness

Idempotency key

Client-supplied unique token that allows a server to safely deduplicate retries for non-idempotent operations (e.g., charging a card).

TERM storage

WAL (Write-Ahead Log)

Append-only log of intended changes written before in-place structures are updated. Enables crash recovery by replaying committed intents.

Related Ch 3
TERM storage

B-tree

Balanced tree over sorted keys stored in fixed-size pages. Supports fast point lookups and range scans with predictable IO depth.

Related Ch 3
TERM storage

LSM-tree

Log-Structured Merge tree: writes buffered in memory and flushed as immutable sorted tables; background compaction maintains read performance.

Related Ch 3
TERM storage

SSTable

Immutable, sorted string table on disk used by LSM engines. Supports efficient range reads and compaction merging.

Related Ch 3
TERM storage

Compaction

Background process that merges and rewrites LSM tables to reduce read amplification and reclaim space from tombstones/old versions.

Related Ch 3
TERM storage, performance

Write amplification

Ratio of physical bytes written to logical bytes written. Increased by compaction, replication, indexes, and GC.

Related Ch 3
TERM storage, performance

Read amplification

Extra disk reads required to satisfy a query due to fragmented data, multiple levels/files, or missing locality.

Related Ch 3
TERM storage

Bloom filter

Probabilistic data structure that quickly tests 'definitely not present' vs 'maybe present' to avoid unnecessary disk reads.

Related Ch 3
TERM evolution

Backward compatibility

New code can read old data/messages. Commonly achieved by additive schema changes with defaults.

Related Ch 4
TERM evolution

Forward compatibility

Old code can tolerate new data/messages (unknown fields). Requires encoding rules that preserve unknown fields or safe ignore semantics.

Related Ch 4
TERM evolution

Schema registry

Central system/process that stores schemas and enforces compatibility checks across producers/consumers.

Related Ch 4
TERM replication

Replication lag

Delay between a write being acknowledged at one replica and becoming visible on another. Drives staleness of follower reads.

Related Ch 5
TERM replication

Leader / follower replication

Single leader defines write order; followers replicate the leader‚Äôs log. Simplifies conflict resolution at cost of leader bottleneck/failover complexity.

Related Ch 5
TERM replication

Multi-leader replication

Multiple leaders accept writes and replicate between regions. Improves locality but requires conflict detection/resolution.

Related Ch 5
TERM replication

Conflict resolution

Rule for reconciling divergent concurrent updates: LWW, merge functions, CRDTs, or application-level reconciliation.

Related Ch 5
TERM replication, consistency

Quorum read/write

Technique where clients read/write from multiple replicas such that read and write sets intersect. Improves availability but does not guarantee linearizability by itself.

Related Ch 5 Ch 9
TERM replication

Anti-entropy

Background process to reconcile replicas (e.g., Merkle trees, gossip). Repairs divergence independent of client reads.

Related Ch 5
TERM partitioning

Partition key

Key used to route data to a shard. Must align with dominant query patterns and avoid hotspots.

Related Ch 6
TERM partitioning

Consistent hashing

Mapping of keys and nodes onto a ring to minimize key movement when nodes join/leave. Helps churn, not hotspots.

Related Ch 6
TERM partitioning, performance

Scatter/gather query

Query that touches many shards and must merge results. Latency tends toward the slowest shard (tail amplification).

Related Ch 6
TERM partitioning, performance

Hotspot / skew

Non-uniform access where a small subset of keys dominates traffic. Causes shard overload and poor scaling.

TERM transactions

MVCC

Multi-Version Concurrency Control: keeps multiple versions of data to allow snapshot reads without blocking writers.

Related Ch 7
TERM transactions

Snapshot isolation

Transactions read from a consistent snapshot and write without blocking readers. Prevents many anomalies but allows write skew.

Related Ch 7
TERM transactions

Write skew

Anomaly where two transactions read the same snapshot and make disjoint writes that violate a cross-row invariant.

Related Ch 7
TERM transactions

Serializable isolation

Strongest isolation where outcome matches some serial order. Implemented via locks, SSI, or deterministic ordering.

Related Ch 7
TERM transactions

2PC (Two-Phase Commit)

Distributed transaction protocol: prepare phase then commit/abort. Can block on failures; adds coordination overhead.

Related Ch 7
TERM transactions

Saga

Distributed workflow pattern that breaks a transaction into steps with compensating actions. Higher availability, more complex correctness.

Related Ch 7
TERM distributed

Partial failure

Some nodes/links fail while others continue. Fundamental to distributed systems; must design for uncertainty.

Related Ch 8
TERM distributed

Failure detector

Mechanism to suspect a node is dead (timeouts/heartbeats). In an asynchronous network, detectors are inherently imperfect.

Related Ch 8 Ch 9
TERM distributed

Clock skew

Difference between physical clocks on different machines. Can break timestamp ordering and expiry logic.

Related Ch 8
TERM distributed

Lamport clock

Logical clock that preserves happens-before ordering via counters piggybacked on messages.

Related Ch 8
TERM distributed, replication

Vector clock

Logical clock using per-node counters to detect concurrency vs causality. Useful for conflict detection in multi-writer replication.

TERM consistency

Linearizability

Strong consistency where operations appear to take effect atomically in real time. Requires coordination; sensitive to partitions.

Related Ch 9
TERM consistency

Causal consistency

Guarantee that reads respect cause-effect relationships without enforcing a total order. Often cheaper than linearizability.

Related Ch 9
TERM consensus

Consensus

Agreement among nodes on a value/order despite failures. Foundation for replicated logs, leader election, and coordination services.

Related Ch 9
TERM consensus

Fencing token

Monotonically increasing epoch/term used to prevent a stale leader from issuing commands after failover.

Related Ch 9
TERM batch

Shuffle

Batch operation that redistributes data across workers based on key. Often the dominant cost due to network + disk spill.

Related Ch 10
TERM batch

Straggler

Slow task that dominates job completion time. Causes tail latency in batch pipelines.

Related Ch 10
TERM stream

Watermark

Stream processing notion of progress in event time: boundary after which events are considered 'late'. Trades correctness vs latency.

Related Ch 11
TERM stream

Window

Time-bounded grouping for stream aggregation (tumbling, sliding, session). Requires state retention and late-event handling.

Related Ch 11
TERM stream

Exactly-once processing

Guarantee that each input event affects output state once. Achieved via idempotency, deduplication, or transactional sinks.

TERM architecture

Derived data

Data computed from other data: indexes, caches, materialized views, aggregates. Must be kept consistent with sources.

TERM architecture

Data lineage

Metadata describing how derived data was produced from sources. Enables audits, debugging, and rebuilds.

Architectural trend to separate storage/compute/indexing/query processing into specialized systems connected by logs/pipelines.
