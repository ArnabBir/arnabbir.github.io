Chapter 32 — Threads: Thread Cancellation

Threads share an address space but execute independently. That sounds simple until you deal with memory visibility, lock ordering, cancellation, thread-local state, and resource lifetime. The pthreads API is a toolset; correctness comes from disciplined invariants and a debugging mindset.

This guide emphasizes race-free patterns: ownership, immutability, critical sections, condition variables with while-loops, and safe shutdown (cancellation points + cleanup handlers).

Invariant
Shared memory requires synchronization; data races are undefined behavior.
Gotcha
Condition variables require a predicate loop to avoid lost/spurious wakeups.
Debugging
Use TSan in tests; perf can reveal contention in futex waits.

Mental model

Internalize these bullets. If you can explain them clearly, you can debug most real-world incidents in this area.
  • Threads share memory; races are bugs unless synchronized (mutex/atomics).
  • A mutex protects an invariant; keep critical sections small and consistent.
  • Condition variables wait for predicates; always check predicate in a loop.
  • Cancellation and shutdown need cleanup handlers and ownership discipline.
  • Thread-local state can hide complexity; document lifetime and destructors.
  • Measure contention: futex waits and context switches indicate scalability limits.

Edge cases & invariants

This is where production bugs live. Expand each panel and treat the bullets as test cases.
Correctness model: data races are bugs
  • If two threads access the same memory and at least one is a write, you need synchronization (mutex/atomic).
  • Thread safety is broader than locks: includes signal safety, cancellation safety, and lifetime management.
  • Condition variables require a predicate and a loop. Spurious wakeups are allowed; missed wakeups happen without correct locking.
  • Prefer ownership/immutability when possible: it eliminates entire classes of races.
Cancellation and cleanup
  • Cancellation is cooperative: threads stop at cancellation points depending on type/state.
  • Always pair cancellation with cleanup handlers (pthread_cleanup_push/pop) so locks and resources are released.
  • Design shutdown protocols: stop accepting work, drain queues, signal workers, join, then free shared state.
  • Avoid mixing fork() with threads unless you exec immediately; it is a common source of deadlocks.
Performance notes
  • Lock contention shows up as CPU in futex waits and high context switches. Reduce critical section size and improve locality.
  • Reader-writer locks can starve writers or readers depending on implementation; measure before choosing.
  • Per-thread storage is useful but can hide memory bloat and complicate teardown. Document ownership clearly.
  • Prefer epoll/event loops for I/O concurrency; use threads for CPU-bound parallelism when work units are coarse enough.
Debugging workflow
  • Use thread sanitizers (TSan) for race detection in tests.
  • gdb/rr can inspect thread stacks; `info threads` and `thread apply all bt` are core moves.
  • Look at /proc/<pid>/task for per-thread stats.
  • perf can attribute time to futex and lock contention hot spots.

Code patterns (C / POSIX)

These are compact patterns to internalize. Treat them as starting points: add proper error handling, timeouts, and logging. 
// Condition variable pattern: predicate + while loop
pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  cv = PTHREAD_COND_INITIALIZER;
int ready = 0;

void *worker(void *arg){
  pthread_mutex_lock(&m);
  while(!ready) pthread_cond_wait(&cv, &m);
  pthread_mutex_unlock(&m);
  return NULL;
}

int main(){
  pthread_t t; pthread_create(&t, NULL, worker, NULL);
  pthread_mutex_lock(&m);
  ready = 1;
  pthread_cond_broadcast(&cv);
  pthread_mutex_unlock(&m);
  pthread_join(t, NULL);
}

Production checklist

If you are shipping code in this area, try to satisfy each checklist item explicitly.
  • Define ownership of shared data; protect mutable shared state with mutexes or atomics.
  • Use condition variables with a predicate in a while-loop; handle spurious wakeups.
  • Avoid global locks in hot paths; minimize critical sections and lock contention.
  • Design clean shutdown and ensure all locks/resources are released on cancellation/error.
  • Use sanitizers in CI (TSan/ASan) and keep reproducers small.
