Chapter 40 — Login Accounting

“System information” chapters are about the operating context: identities (uid/gid), time, resource limits, and introspection. These APIs are where portability surprises and security boundaries show up in real programs.

Production workflows depend on observability: /proc, rlimits, uname/sysinfo, and careful error messages that carry enough context to debug failures in the field.

Invariant
Effective IDs + rlimits shape what your process can do.
Gotcha
sysconf can return -1 for both error and 'unlimited'—check errno.
Debugging
Use /proc and prlimit/ulimit to inspect limits and runtime state.

Mental model

Internalize these bullets. If you can explain them clearly, you can debug most real-world incidents in this area.
  • Environment matters: ids, groups, rlimits, and time sources change behavior.
  • Effective IDs control permissions; supplementary groups expand access.
  • rlimits cap resources and often cause production failures (too many open files).
  • /proc is a live API for debugging: fds, maps, status, and more.
  • Clock selection matters: monotonic for intervals, realtime for timestamps only.
  • Make errors actionable by logging context and current limits.

Edge cases & invariants

This is where production bugs live. Expand each panel and treat the bullets as test cases.
Identity and credentials
  • Linux tracks multiple UIDs/GIDs: real, effective, and saved. The effective IDs control permission checks.
  • Supplementary groups widen access; group lookup APIs are often non-thread-safe unless *_r variants are used.
  • Environment variables can be part of the trust boundary — especially for privileged programs.
  • Always log IDs and key environment for debugging authz failures in production.
Resource limits and introspection
  • rlimits cap resources like open files and address space. Many production incidents are simply RLIMIT_NOFILE too low.
  • sysconf/pathconf query system limits. Be careful: some return -1 for “no limit” and also on error (check errno).
  • /proc is a live view of kernel state: fd links, maps, status, and cgroup placement.
  • Time APIs: choose CLOCK_MONOTONIC for measuring intervals; realtime can jump due to NTP/time adjustments.
Debugging workflow
  • First confirm effective IDs (id -a, /proc/<pid>/status).
  • Inspect rlimits (ulimit -a, prlimit).
  • Use /proc for process-specific insight; it’s often faster than adding logs.
  • For time bugs, log clock source, timeouts, and monotonic deltas.

Production checklist

If you are shipping code in this area, try to satisfy each checklist item explicitly.
  • Log effective IDs, rlimits, and key environment at startup for debuggability.
  • Use CLOCK_MONOTONIC for timeouts; never rely on realtime for deadlines.
  • Handle “unlimited” returns from sysconf/pathconf correctly (check errno).
  • Treat /proc as the source of truth for runtime state; build tooling around it.
  • Make errors actionable: include which resource, which limit, and current values.
