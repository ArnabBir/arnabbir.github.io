Chapter 62 — Terminals

Terminals are a historical interface that still shapes Unix UX: line disciplines, canonical vs raw mode, job control, and controlling terminals. Pseudoterminals (pty) are the modern workhorse behind SSH, terminal emulators, containers, and expect-like automation.

This guide focuses on how the kernel routes bytes through tty layers and how process groups interact with signals (SIGINT/SIGTSTP) during interactive control.

Invariant
TTY driver mediates input editing and generates job-control signals to foreground pgrp.
Gotcha
If you don't restore termios modes, you can break the user's shell.
Debugging
Use stty -a and strace for tcsetattr/ioctl; inspect process groups.

Mental model

Internalize these bullets. If you can explain them clearly, you can debug most real-world incidents in this area.
  • Terminals are stateful devices with line disciplines; modes change behavior radically.
  • Job control signals are generated by tty driver and sent to foreground process group.
  • Raw mode is required for full-screen apps; always restore settings on exit.
  • PTYs emulate terminals: master controlled by emulator, slave seen by program.
  • Interactive I/O is timing-sensitive; handle EAGAIN/EINTR and partial reads.
  • Controlling terminal presence/absence changes semantics in daemons/containers.

Edge cases & invariants

This is where production bugs live. Expand each panel and treat the bullets as test cases.
Terminal modes
  • Canonical mode buffers input until newline; line editing happens in the kernel line discipline.
  • Raw mode delivers bytes immediately (no echo, no line editing) — used by full-screen apps and SSH.
  • Signals like SIGINT (Ctrl-C) are generated by the tty driver and delivered to foreground pgrp.
  • termios controls this behavior; changes are per-tty and can affect child processes.
Job control mechanics
  • Foreground/background is managed via process groups attached to a controlling terminal.
  • When a background process writes to terminal, it may receive SIGTTOU depending on settings.
  • Shells orchestrate setpgid + tcsetpgrp and handle SIGCHLD to manage jobs.
  • Containers often lack a controlling terminal; tools behave differently.
PTYs in real systems
  • A pty pair has a master and slave. Terminal emulators and SSH use the master; the program sees the slave as a tty.
  • PTYs enable automation tools (expect), multiplexers (tmux), and remote sessions.
  • Be careful with buffering and EIO behavior when the master disappears.
  • Security: grantpt/unlockpt control ownership and access.
Debugging workflow
  • Use stty -a to inspect settings. Use strace to observe tcsetattr/ioctl calls.
  • Check controlling terminal and pgrp via /proc/<pid>/stat and /proc/<pid>/fd/0 links.
  • When interactive programs misbehave, suspect terminal mode not restored (crash without cleanup).
  • Always restore termios on exit (atexit or signal handling).

Production checklist

If you are shipping code in this area, try to satisfy each checklist item explicitly.
  • Always restore terminal modes on exit and on signals; otherwise you break the user’s shell.
  • Treat job control signals as part of your control plane (SIGINT/SIGTSTP/SIGHUP).
  • Prefer ptys for interactive subprocess control; document master/slave responsibilities.
  • Handle EIO/hangup behavior when the controlling side disappears.
  • Log termios settings when debugging weird interactive behavior.
