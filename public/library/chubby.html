01. Sessions & KeepAlives

A Chubby session is maintained by periodic KeepAlives. If the Master fails, the client enters a "Jeopardy" state (Grace Period). If a new Master is elected before the grace period expires, the session is saved.

Master Lease

The Master promises to not unilaterally terminate the session for a specific interval.

Jeopardy (Grace Period)

When the client's local lease timer expires, it disables its cache and waits (45s default). If communication restores, it resumes. If not, session expires.

KeepAlive Blocking

The Master blocks KeepAlive RPCs until the client's lease is close to expiring. This allows the Master to "push" events and cache invalidations by returning the blocked call early.

02. Locks & Sequencers

Advisory Locks

Chubby locks are advisory (like mutexes). They do not inherently stop other clients from accessing the resource. Holding a lock F doesn't prevent others from reading F.

Client A: Acquire("/ls/foo/lock") -> OK
Client B: Acquire("/ls/foo/lock") -> BLOCK

The Zombie Client Problem

What if a client holding a lock stalls (GC pause), loses the lock, but then wakes up and tries to write? Chubby solves this with Sequencers.

1. Client A: GetSequencer() -> "Lock:42"
2. Client A: Sends Write(Data, "Lock:42")
... A stalls ... Lock expires ... B acquires (Lock:43) ...
3. Server: Receives Write. Checks 42 < 43.
4. Server: REJECT.
Servers can validate sequencers against their Chubby cache or simply reject any request with a number less than the highest they've seen.

03. Consistent Caching

Write-Through Caching

Chubby clients cache file data and meta-data. The cache is consistent. Before a file can be modified, the Master must invalidate cached copies on all clients.

  • Reduces read traffic on Master.
  • Write blocked until invalidation ACKs received.
  • Events (invalidation) piggybacked on KeepAlive replies.
  • Open Handle Caching: Clients cache open handles, so repeated opens of the same file are local and virtually free.
Client Cache Protocol
1. Master receives Write --->
2. Send Invalidate(File) ---> Client
3. Client Flushes Cache [Empty]
4. Client ACKs (KeepAlive) <---
5. Master performs Write [Commit]
