Appendix E — Further Sources of Information

This appendix is your map for going deeper than TLPI. When you hit a subtle behavior or performance issue, you need primary sources: kernel docs, libc docs, and reproducible experiments.

Use this as a checklist of where to look depending on the kind of question you have (API semantics vs performance vs security).

Invariant
Syscalls expose kernel objects; libc wraps them and sets errno on failure.
Gotcha
Man pages have details in NOTES/CAVEATS that are easy to miss.
Debugging
Use strace early; it quickly reveals the kernel boundary and errors.

Primary sources
  • man 2/3 and man7.org for syscall/libc behavior.
  • Linux kernel documentation (in the kernel tree) for subsystem internals.
  • glibc manual for libc-level semantics (stdio, threads, locale).
  • POSIX/SUS docs for portability requirements and defined behavior.
Performance & tracing
  • perf: profiling and kernel stacks.
  • eBPF/BCC/bpftrace: dynamic instrumentation and latency debugging.
  • ftrace/tracefs: kernel-level function and event tracing.
  • Flame graphs for visualizing CPU stacks.
Security & hardening
  • Capabilities and securebits: model privilege transitions.
  • seccomp for syscall filtering.
  • Namespaces + cgroups for isolation and resource control.
  • Audit logs (if enabled) for security-relevant events.

API quick reference

APIs you’ll likely use while working through these topics.
API Signature / shape Common errno Notes
perrorvoid perror(const char *s);—Prints s + error message for current errno to stderr.
errnoextern int errno; (thread-local)—Thread-local error indicator set by failing syscalls/libc wrappers; read immediately after failure.
strerrorchar *strerror(int errnum);—Maps errno value to human-readable string; strerror_r() is safer in threads.

Production checklist

Checklists are how you stop repeating incidents.
  • Always read the ERRORS section of man pages and treat it as part of your API contract.
  • Write minimal experiments to confirm assumptions; keep strace/perf in your toolbox.
  • Separate portability layer from Linux-specific optimizations.
  • Make error messages actionable: include inputs and context.
  • Prefer simple, explicit invariants over clever code.
