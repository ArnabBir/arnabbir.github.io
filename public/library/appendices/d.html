Appendix D — Kernel Configuration

Kernel configuration shows up in real debugging constantly: sysctls, /proc tunables, limits, and feature availability. Many “my code is broken” incidents are actually “the kernel is configured differently than you assumed”.

This appendix gives you a practical map of where configuration lives and how to reason about it safely.

Invariant
Effective IDs + rlimits shape what your process can do.
Gotcha
sysconf can return -1 for both error and 'unlimited'—check errno.
Debugging
Use /proc and prlimit/ulimit to inspect limits and runtime state.

Where configuration lives
  • /proc/sys/*: sysctl interface (runtime tunables). Example: net.core.somaxconn influences listen backlog behavior.
  • /sys/*: device and subsystem configuration surfaces (cgroups, block devices, network).
  • /boot/config-$(uname -r): build-time kernel config (if installed).
  • sysctl(8) tool provides a stable interface to /proc/sys.
Operational safety rules
  • Treat configuration as part of your deployment contract. Document required sysctls and validate at startup.
  • Prefer per-service isolation (namespaces/cgroups) over global sysctls when possible.
  • Never change sysctls in ad-hoc ways on production systems without rollback and monitoring.
  • Understand scope: some sysctls are per-namespace; others are global.
Debugging patterns
  • If networking behaves oddly, check somaxconn, tcp_tw_reuse, and related tunables.
  • If memory behavior is weird, inspect overcommit and cgroup memory settings.
  • If file descriptor usage fails, check rlimits and fs.file-max.
  • Always record kernel version and config when collecting incident evidence.

# The raw files are in /proc/sys (dots become slashes)
cat /proc/sys/net/core/somaxconn

Production checklist

Checklists are how you stop repeating incidents.
  • Log effective IDs, rlimits, and key environment at startup for debuggability.
  • Use CLOCK_MONOTONIC for timeouts; never rely on realtime for deadlines.
  • Handle “unlimited” returns from sysconf/pathconf correctly (check errno).
  • Treat /proc as the source of truth for runtime state; build tooling around it.
  • Make errors actionable: include which resource, which limit, and current values.
