Appendix C — Casting the NULL Pointer

This appendix is about C-level correctness: how NULL works, what pointer casts mean, and how to avoid undefined behavior when interfacing with low-level APIs.

In systems code, a surprising number of bugs are type/ABI mismatches: passing the wrong pointer type, assuming NULL is always 0 in all contexts, or using integer sentinel values incorrectly.

Invariant
Syscalls expose kernel objects; libc wraps them and sets errno on failure.
Gotcha
Man pages have details in NOTES/CAVEATS that are easy to miss.
Debugging

Deep dive

Practical panels: tools, workflows, and invariants worth memorizing.
What is NULL?
  • NULL is a null pointer constant used to represent 'points to nothing'. In C it is typically defined as ((void*)0) or 0 depending on headers and language mode.
  • A null pointer is not required to have all-bits-zero representation on every architecture (though it usually does on modern Linux).
  • Use NULL for pointers and 0 for integer/size values; this makes intent clear and avoids overload confusion.
Casting and API boundaries
  • Avoid unnecessary casts. In C, void* implicitly converts to/from object pointers; casting can hide bugs.
  • When a syscall expects a pointer (e.g., struct sockaddr*), pass the correct type; don't cast arbitrary integers.
  • Be careful with varargs functions: type mismatches are not checked by the compiler and can corrupt the call frame.
  • In C++, use nullptr instead of NULL to avoid integer/pointer ambiguity.
Practical discipline
  • Compile with -Wall -Wextra -Werror and enable warnings like -Wconversion in strict codebases.
  • Use sanitizers (UBSan) to catch undefined behavior early.
  • Document sentinel values: if you use (void*)-1 as an error sentinel (common with mmap), treat it consistently and check errno.

Production checklist

Checklists are how you stop repeating incidents.
  • Always read the ERRORS section of man pages and treat it as part of your API contract.
  • Write minimal experiments to confirm assumptions; keep strace/perf in your toolbox.
  • Separate portability layer from Linux-specific optimizations.
  • Make error messages actionable: include inputs and context.
  • Prefer simple, explicit invariants over clever code.
