Definitions

Use these terms consistently to reduce ambiguity and prevent incorrect “it seems safe” reasoning.

Terms

reference
TermDefinition
AtomicityAn operation is atomic if it appears indivisible: other threads cannot observe intermediate states. Atomicity is about interleavings; it often requires locks or atomic instructions.
VisibilityA write is visible to another thread when that thread is guaranteed to observe the write (or a later write). Visibility requires a happens-before relationship.
Happens-before (HB)A partial order describing when actions in one thread must be observed by another. HB edges come from program order, locks, volatile, start/join, etc; HB is transitive.
Data raceTwo accesses to the same variable from different threads, at least one a write, without synchronization. Data races make behavior unpredictable under the memory model.
InvariantA property that must always hold for the state of a component (e.g., queue size matches links). Concurrency control must preserve invariants under interleavings.
LinearizabilityA correctness condition for concurrent objects: each operation appears to take effect instantaneously at some point between invocation and response, consistent with real-time ordering.
DeadlockA cycle in the waits-for graph: each participant waits for a resource held by another. Classic necessary conditions include mutual exclusion, hold-and-wait, no preemption, and circular wait.
StarvationA thread makes no progress because it never gets scheduled or never acquires a resource due to unfairness or priority inversion.
LivelockThreads keep running and responding (e.g., retrying) but make no forward progress; often a symptom of contention + naive backoff.
BackpressureA mechanism that prevents overload from turning into unbounded queues and latency collapse by forcing producers to slow down or shed load.
CAS (compare-and-set)An atomic primitive that updates a value if it matches an expected value. CAS underlies many lock-free algorithms but can suffer ABA without versioning.
Lock-freeA progress guarantee: in a finite number of steps, some thread will complete. Individual threads may starve.
Wait-freeA stronger guarantee: every thread completes in a finite number of steps.
Safe publicationPublishing an object reference such that other threads are guaranteed to see the fully constructed object state.

Precision in words → precision in invariants → fewer concurrency incidents.
