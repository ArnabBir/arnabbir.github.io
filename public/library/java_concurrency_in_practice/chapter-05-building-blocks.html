Concurrent building blocks: choose the right primitive

toolbox

The platform gives you pre-built pieces whose correctness is carefully engineered: concurrent collections, blocking queues, and synchronizers. Your job is to map your problem’s invariants and progress requirements to the right building block.

Producer–consumer with BlockingQueue

pattern
// Producer–consumer with bounded queue BlockingQueue<Job> q = new ArrayBlockingQueue<>(100); void producer() throws InterruptedException { while(true){ Job j = makeJob(); q.put(j); // blocks when queue is full -> backpressure } } void consumer() throws InterruptedException { while(true){ Job j = q.take(); // blocks when queue is empty handle(j); } }

Blocking queues give you a handoff point that is both thread-safe and a natural place to implement backpressure.

Synchronizers: shaping concurrency

sync

Synchronizers coordinate multiple threads around a condition:

  • Semaphore: limit concurrency (e.g., max N in-flight requests).
  • CountDownLatch: one-time gate (wait for N events).
  • CyclicBarrier: repeated rendezvous (phased computations).
  • Exchanger: pairwise handoff.

Engineering note: these primitives are correctness tools first; throughput is a bonus. Choose the one whose state machine matches your problem.

Caches and memoization: the “check-then-act” trap

pitfall
// Broken: race + redundant work V get(K k){ V v = map.get(k); if(v == null){ v = compute(k); map.put(k, v); } return v; }

This is a classic compound action: multiple threads can compute and overwrite values. Fix it with computeIfAbsent, memoizers, or a lock around the whole transition.
