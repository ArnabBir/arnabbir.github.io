Cancellation is a protocol

lifecycle

Cancellation is not a “kill switch”; it’s a cooperative protocol between:

  • the caller (who requests cancellation),
  • the task (which checks for cancellation),
  • the executor/thread (which delivers interruption).

Well-designed code makes cancellation prompt, safe, and predictable.

Interruption policy: your team needs one

best practice
// Example: interruption-aware loop while(!Thread.currentThread().isInterrupted()){ try{ doBlockingWork(); // e.g., take/put/sleep/IO } catch(InterruptedException e){ Thread.currentThread().interrupt(); // restore status break; // exit promptly } } cleanup();

Rule: if you catch InterruptedException and can’t fully handle it, reassert the interrupt and exit/propagate.

Shutdown: design for it early

systems

Robust services have explicit lifecycle methods: start(), stop(), and awaitTermination(). “Fire-and-forget” threads make shutdown unreliable and leak resources.

Common approach: a coordinator thread signals shutdown, tasks stop accepting new work, and in-flight tasks either complete within a timeout or are cancelled.

Poison pills and end-of-stream

pattern

In producer-consumer designs, a sentinel element (poison pill) can signal consumers to stop. The key is to ensure every consumer observes termination:

  • enqueue N poison pills for N consumers, or
  • use a shared “done” flag + interruption, or
  • close the queue/channel if your abstraction supports it.
