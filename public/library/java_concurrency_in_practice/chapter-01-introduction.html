Threads, parallelism, and the shape of modern software

foundation

Threads are a resource multiplexing mechanism: the CPU time slice and memory model expose interleavings that your code must tolerate. You use concurrency for:

  • Responsiveness: keep a UI/event-loop responsive by pushing long work off the event thread.
  • Throughput: overlap I/O and compute; keep cores busy by working on independent tasks.
  • Latency hiding: issue multiple requests concurrently, then compose results.

The trap: concurrency doesn’t just reorder operations; without the right synchronization it can change what values are observable, because compilers/CPUs can reorder and cache aggressively.

Safety vs liveness: two orthogonal correctness axes

mental model

Safety properties are invariants: “nothing bad happens”. Examples: counters never go negative, map never loses an entry, money is conserved. Safety failures are often race conditions (lost updates, broken invariants, stale reads).

Liveness properties are progress guarantees: “something good eventually happens”. Examples: request eventually completes, threads don’t get stuck forever. Liveness failures include deadlock, livelock, and starvation.

Engineering rule: design correctness as state + transitions. Specify the invariant, then choose a coordination primitive that makes each transition atomic with respect to the invariant.

A practical workflow for building intuition

how to study

1) Identify shared mutable state. 2) Decide the synchronization policy (lock? immutability? confinement? atomics?). 3) Prove the invariant under interleavings. 4) Check visibility (publication & happens-before). 5) Stress-test with adversarial schedules.

★
Gold-standard mental picture: a method is safe when its externally visible behavior is equivalent to some serial execution order (linearizability) for the operations that matter.
