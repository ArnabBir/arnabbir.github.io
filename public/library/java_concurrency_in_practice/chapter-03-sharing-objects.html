Visibility is a correctness issue, not a performance detail

memory

Even if an operation looks “atomic”, other threads may not see it unless a happens-before relationship is established. Without sufficient synchronization, a thread may read a stale cached value, or observe writes out of order.

Publication is the act of making an object reference reachable by another thread. Publication is safe only when the object’s construction is visible to the reading thread.

Four strategies for avoiding shared mutability

design

1) Thread confinement: keep data private to a thread (locals, thread-owned actors, thread-local storage).

2) Immutability: publish an immutable object; no writes after publication.

3) Synchronization: guard state with a lock and enforce a lock discipline.

4) Atomic/lock-free: represent the shared state as an atomic variable and update via CAS.

Rule of thumb: if you can make data immutable or confined, do that first; it removes entire classes of bugs.

Safe publication checklist

checklist
  • Publish via static initializer or final static fields.
  • Publish via volatile reference or atomic reference.
  • Publish under a lock (write and read under same lock).
  • Use thread-safe collections / concurrent queues for handoff.
  • Avoid “this escapes” during construction: don’t start threads or register callbacks from constructors unless you can guarantee they won’t run until construction completes.
