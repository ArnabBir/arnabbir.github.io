Liveness hazards: progress can fail even when safety holds

liveness

Your invariant might be perfectly protected, yet the system can still fail by making no progress. Classic hazards:

  • Deadlock: cyclic waiting for locks/resources.
  • Starvation: a thread never gets CPU or lock access due to unfair scheduling or priority inversion.
  • Livelock: threads keep “doing something” (retries/rollbacks) but make no progress.

Diagnosis toolkit

operations

When production stalls:

  • Capture thread dumps; look for BLOCKED / WAITING states.
  • Enable lock contention monitoring (where available).
  • Add timeouts and telemetry around lock acquisition and queue waits.

Design-time fix: minimize nested locks, define lock ordering, and prefer higher-level concurrency utilities that avoid manual lock choreography.

Avoiding deadlock in practice

rules

Three practical strategies:

1) Lock ordering: assign each lock a rank and always acquire in increasing order.

2) Try-lock with timeout: if you can’t acquire quickly, back off and retry in a safe order.

3) Open calls: don’t call external code (callbacks, remote services) while holding locks.
