Building custom synchronizers: be careful

expert

Custom synchronizers are essentially small distributed protocols. Correctness requires a clear state machine, safe publication, and robust handling of interruption and timeouts.

In practice, prefer built-in synchronizers unless you have a strong reason (performance characteristics, bespoke semantics, or educational exploration).

AQS mental model (high-level)

aql

AbstractQueuedSynchronizer (AQS) is a framework for building locks and synchronizers. It manages:

  • a single state integer (or bits) representing the synchronizer’s state
  • a FIFO wait queue of threads
  • acquire/release mechanics that park/unpark threads

Design trick: encode your synchronizer’s state so that the transition “tryAcquire” can be expressed as a CAS loop on that single state variable.

Condition queues: the predicate loop

correctness
// The correct waiting pattern lock.lock(); try{ while(!predicate()){ condition.await(); } // predicate holds; do work } finally { lock.unlock(); }

Always wait in a loop. Wakeups can be spurious, and signals can be lost if you don’t re-check the predicate under the lock.

When to build a custom synchronizer

decision

Good reasons include: nonstandard admission control, specialized fairness constraints, or needing to coordinate many threads around a compact shared state. Bad reasons: “because it’s fun” in production code.
