Definition of thread safety (the spec, not the implementation)

fundamentals

A component is thread-safe if it behaves correctly when accessed from multiple threads, without requiring callers to do extra synchronization. “Correctly” means it preserves its invariants and meets its postconditions under all interleavings.

Thread safety is a property of the contract: thread-safe classes must document their synchronization policy (what’s guarded, by which lock, and what callers may assume).

Guarding state with locks: granularity, scope, and invariants

locks

Locks are about making a compound action atomic with respect to an invariant. The lock must cover every read and write that participates in the invariant, and the invariant must not be observable in a broken intermediate state.

  • Lock granularity: coarser locks simplify correctness but increase contention.
  • Lock scope: keep critical sections small; don’t hold locks across I/O or callbacks.
  • Lock ordering: define a total order to prevent deadlock when acquiring multiple locks.

Atomic variables: when they’re enough (and when they’re not)

atomics

Atomic variables solve a narrower problem: they make a single variable update atomic and provide visibility similar to volatile for the variable’s value. They’re ideal for:

  • independent counters and sequence generators
  • simple state machines expressed as one variable
  • nonblocking algorithms where the invariant can be maintained by updating one pointer at a time

If you need to update multiple fields together (e.g., balance and history), atomics alone are usually insufficient—you need a lock or a broader atomic representation.
