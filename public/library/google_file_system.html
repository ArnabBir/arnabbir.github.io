DISTRIBUTED LOCKING

Namespace Management

Unlike traditional file systems, GFS does not maintain a directory tree data structure. It maps full pathnames to metadata in a massive lookup table. To ensure consistency during concurrent operations (like creating a file while deleting its parent), the Master applies a strict locking protocol on paths.

Locking Protocol Simulator

Locking Rules:

  • Ancestors: Acquire Read Lock (Shared). Prevents deletion/rename of parent directories.
  • Target: Acquire Write Lock (Exclusive) on the specific leaf node.
  • Locks are acquired in strictly consistent path order to prevent deadlocks.

01. Architecture Mechanics

Single Master Strategy

To simplify design, GFS uses a single master. To prevent bottlenecks, clients never move data through the Master.

  • Holds file system metadata in RAM.
  • Manages Leases and Garbage Collection.
  • Single point of failure (mitigated by Shadow Masters).

The 64MB Chunk

Files are divided into fixed-size 64MB chunks. This is massive compared to standard OS block sizes (4KB).

Why so large?

  1. Reduces Master's metadata size (kept in RAM).
  2. Reduces client-master interaction.
  3. Persistent TCP connection to chunkservers.

Relaxed Consistency

GFS relaxes consistency for performance. It distinguishes between Consistent and Defined states.

// Atomic Record Append if (write > 64MB) { pad_chunk(); retry_next_chunk(); } else { append_at_offset(); }

02. The Consistency Model

GFS relaxes consistency to achieve high aggregate throughput. It distinguishes between a Defined state (all clients see same data, fully written) and a Consistent state (all clients see same data, but it might be partial/undefined).

Operation Write Success Concurrent Write Success Failure
Region State Defined Consistent (Undefined) Inconsistent
Explanation All replicas identical. Data is what app wrote. All replicas identical, but data from multiple writers is interleaved. Replicas differ. Client sees different data.

Atomic Record Appends

Unlike standard writes where the client specifies the offset, in GFS appends, the Primary chooses the offset. If an append fails at a secondary, the client retries. This creates "padding" (garbage) regions in the file, but ensures that the data block exists at least once at the same offset on all replicas.

Garbage Collection

GFS does not reclaim physical storage immediately when a file is deleted.

// 1. Rename to hidden file rename("foo", ".trash/foo_timestamp"); // 2. Master periodic scan if (now() > file.deletion_time + 3d) { remove_metadata(file); } // 3. Stale Replica Detection // During Heartbeat if (chunk.version < master.version) { chunkserver.delete(chunk); }
