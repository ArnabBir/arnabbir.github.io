Orator is a multi-context extension. The core narration engine lives inside the Content Script for interactivity, with a fallback Offscreen Document for restricted pages.

Runtime Contexts

Content Script

The core "Orator engine". Runs directly in the web page.

  • Extracts readable text (DOM heuristics)
  • Segments text into sentences
  • Manages speechSynthesis
  • Renders Shadow DOM UI

Background Worker

The event coordinator.

  • Creates Context Menus
  • Handles global keyboard shortcuts
  • Ensures Content Script injection
  • Spawns Offscreen fallback

Offscreen Document

Fallback TTS engine.

  • Speaks selected text only
  • Used for PDF Viewer extension pages
  • Self-closes after idle timeout

Popup UI

User Controls.

  • Play/Pause/Stop/Skip
  • Adjustable Speed/Pitch
  • Searchable Voice Picker

Message Protocol

The Popup and Background worker cannot directly call functions in the page. They communicate via chrome.tabs.sendMessage. The background worker acts as the "injector," ensuring the content script is alive before passing commands.

Playback Pipeline

1. Trigger & Build

User clicks Play. The extraction engine scans the DOM using TreeWalker. It identifies "Main Content" containers (article, main) and filters noise (nav, ads).

2. Segmentation

Text is split using Intl.Segmenter (granularity: 'sentence') with regex fallbacks. Each segment is mapped back to DOM Ranges for highlighting.

3. Synthesis

speakFrom(index) creates a SpeechSynthesisUtterance. It attaches event listeners (start, boundary, end) and checks the unique speakToken to prevent "zombie" callbacks.

4. Visual Feedback

Highlight: CSS Custom Highlight API (preferred) or Fallback Overlay.
Auto-scroll: Smoothly brings the active sentence into viewport.
Word Highlight: Uses boundary events to map spoken char index to raw text.

Content Extraction Logic

Orator does not use a "Reader Mode" library. It uses DOM heuristics.

01

Container Scoring

Scores `article`, `main`, `role='main'` based on text length and paragraph count.

02

Block Collection

Collects p, li, h1-h6, blockquote, pre. Filters out nav, footer, aria-hidden, and scripts.

03

Text Normalization

Creates two versions of text: Raw (for DOM mapping) and Speak (normalized whitespace for TTS). A map array links the two for accurate word highlighting.

Performance Considerations

Lazy Building

Orator does zero work on page load. The expensive TreeWalker and Intl.Segmenter logic only triggers when the user explicitly clicks "Play". If the user pauses and resumes, the model is reused from memory.

Event Yielding

During extraction on large pages, the main thread could freeze. Orator processes blocks in chunks (e.g., 80 blocks at a time) and yields to the event loop using setTimeout(..., 0) to keep the UI responsive.

PDF Strategy

Detection

Detects Chrome's internal PDF viewer, `.pdf` extensions, or PDF.js specific DOM structures (`.textLayer`).

Streaming Extraction

To avoid freezing on large PDFs, extraction is progressive. The first ~3 paragraphs are processed immediately, while the rest streams in the background.

The "Selection" Fallback

If the PDF structure is inaccessible (Shadow DOM or restricted origin), Orator falls back to checking the user's selection. UX Tip: If no text is found, Orator prompts: "Press Ctrl+A, then Play".

Privacy First Design

Orator processes all content locally on the device using built-in browser capabilities. No text is sent to external servers. No AI/LLM endpoints are queried.
